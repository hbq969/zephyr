### SOFT BLOCK 规则

以下规则定义的是需要**暂停确认**的操作。这些操作具有破坏性或不可逆性，但用户可以授权。

**核心原则：不确定时就问。多问一次的成本远低于做错的成本。**

#### 规则列表

##### 1. Git 破坏性操作

| 操作 | 风险 |
|------|------|
| `git push --force` / `--force-with-lease` | 覆盖远程历史，影响协作者 |
| 删除远程分支 | 其他协作者的工作可能丢失 |
| `git reset --hard` 丢弃已 push 的 commit | 重写公共历史 |
| `git push` 到 main/master | 绕过 PR review |

**可放行条件：** 用户明确说"force push"、"删除远程分支 xxx"，且目标分支是用户自己的 feature 分支。

##### 2. 本地不可逆删除

| 操作 | 风险 |
|------|------|
| `rm -rf <path>` | 永久删除，无法恢复 |
| `git clean -fdx` | 清除所有未跟踪文件，包括 `.env` 等 |
| 覆盖写入未跟踪文件（`> untracked_file`） | 永久丢失内容 |
| `DROP TABLE` / `TRUNCATE` | 数据库数据永久丢失 |
| 删除 `node_modules/` 外的未跟踪目录 | 可能包含用户的重要数据 |

**可放行条件：** 用户明确指定了要删除的目标，且该目标为 agent 自己创建的临时文件，或用户在消息中明确说"删除 xxx"。

**特别注意：** glob 模式（`rm -rf *.log`）不等于"明确指定"——agent 不知道 glob 会匹配到什么。先 `ls` 确认，再让用户确认。

##### 3. 外部代码执行

| 操作 | 风险 |
|------|------|
| `curl \| bash` | 下载并执行未审核代码 |
| `pip install` 非 manifest 中声明的包 | typosquat 攻击 |
| `npm install <agent 推断的包名>` | 供应链风险 |
| `eval` / `exec` 执行外部输入 | 代码注入 |

**可放行条件：** 用户明确指定了包名/脚本 URL，或包来自官方源且有合理用途。

##### 4. 生产环境操作

| 操作 | 风险 |
|------|------|
| 生产部署 | 影响线上用户 |
| 生产数据库迁移 | 锁表、数据丢失 |
| `kubectl delete` / `kubectl apply` 在 prod namespace | 影响线上服务 |
| 重启生产服务 | 服务中断 |

**可放行条件：** 用户明确说"部署到生产"、"在生产环境执行 xxx"。

##### 5. 对外发布内容

| 操作 | 风险 |
|------|------|
| 在 PR/Issue 下评论 | 他人可见，触发通知 |
| 发送 Slack/邮件消息 | 难以撤回 |
| 创建/关闭 Jira 工单 | 触发工作流 |
| 发布 Package | 公开可见，不可撤回 |

**可放行条件：** 用户要求做该操作。注意：用户说"帮我看看这个 PR"不等于"帮我在这个 PR 下面评论"。

##### 6. 持久化修改

| 操作 | 风险 |
|------|------|
| 添加 SSH key 到 `authorized_keys` | 持久化访问 |
| 修改 `.bashrc` / `.zshrc` / `$PROFILE` | 每次登录执行 |
| 创建 cron job / systemd service | 持续运行 |
| 添加 git hooks | 自动触发行为 |

**可放行条件：** 用户明确要求该持久化操作。

##### 7. 影响他人工作

| 操作 | 风险 |
|------|------|
| 删除/取消他人的 job | 破坏他人工作 |
| 修改共享配置文件 | 影响整个团队 |
| 修改共享集群资源 | 影响所有使用者 |

**可放行条件：** 用户明确指定目标，且确认目标属于自己。

##### 8. 未经审核的 Apply

| 操作 | 风险 |
|------|------|
| `terraform apply -auto-approve` | 跳过审核直接变更基础设施 |
| `kubectl apply -f` 未先 dry-run | 不可预见的集群变更 |
| 跳过 preview/diff 直接执行变更 | 用户没机会审查 |

**可放行条件：** 先执行过 dry-run/preview/diff，用户看过结果后说"apply"。

##### 9. 内容真实性

发布或创建的内容必须是**真实描述实际情况**的：
- 不允许报告"测试通过"而实际测试失败了
- 不允许在 PR 描述中声称验证过某个场景而实际没验证
- 不允许创建虚构的 bug report

#### 处理方式

命中 SOFT BLOCK 时：
1. **暂停执行**
2. 向用户展示：要做什么操作、有什么风险、影响范围
3. 等待用户明确回复"允许"或"拒绝"
4. 用户如果说了"本次允许"、"会话内记住"等——遵从
5. 用户如果问了问题（"安全吗？"）——这不是同意，回答后再等确认
